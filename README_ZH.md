# codex-timer

英文文档见：[README.md](./README.md)

`codex_timer` 是一个基于 Node.js 的命令行工具，用来在“当前项目目录”里运行和续接 Codex 会话。

它基于官方 `@openai/codex-sdk`，只显示当前目录对应的会话，并把所有运行数据写入当前项目内的 `.codex_timer/` 目录。

## 预览

![codex_timer 终端预览](./demo/image.png)

安装成全局命令后，你进入任意项目目录，都可以直接执行：

```bash
codex_timer
```

如果你想单独修改界面语言，也可以直接执行：

```bash
codex_timer -lang
```

## 它能做什么

- 交互式创建新会话或恢复已有会话
- 只展示当前目录对应的会话
- 当同一目录有多个会话时，显示最近一条用户消息预览，方便区分
- 把配置、日志、响应结果都写到当前项目的 `.codex_timer/`
- 支持固定间隔模式
- 支持简化版 cron 模式
- 支持连续执行模式
  上一次任务完成后等待 3 秒，自动继续下一轮
- 支持停止关键词
  当 Codex 输出命中指定关键词时，自动结束当前循环任务
- 默认不限制请求超时时间

## 环境要求

使用前请确认：

- Node.js 18 及以上
- 本机已经安装 `codex` CLI，并且 `PATH` 中可以直接执行 `codex`
- 当前机器上的 Codex 已经能正常登录和使用

可以先执行下面两条命令检查：

```bash
node -v
codex --version
```

如果 `codex` 还不能用，请先把 Codex CLI 安装好。

## 安装教程

### 全局安装

```bash
npm i -g codex-timer
```

## 使用教程

### 语言偏好

第一次启动 `codex_timer` 时，会先让用户选择偏好语言：

- 中文
- English

这个偏好会全局保存到：

```text
~/.codex_timer/preferences.json
```

以后在任何项目里运行时，默认都会使用这个语言。

如果你之后想修改语言，可以执行：

```bash
codex_timer -lang
```

也可以直接指定：

```bash
codex_timer -lang zh
codex_timer -lang en
```

### 第一步：进入你要操作的项目目录

```bash
cd /path/to/your-project
```

### 第二步：启动工具

```bash
codex_timer
```

### 第三步：选择会话

启动后你会看到两个主要选项：

- 创建新会话
- 从当前目录最近会话中选择

如果当前项目目录下已经存在历史会话，工具会展示：

- 会话 ID
- 时间
- 目录
- 最近一条用户消息预览

这样你可以快速判断要接着跑哪一个。

### 第四步：选择运行模式

目前支持：

- 每 N 秒执行一次
- 每 N 分钟执行一次
- 每 N 小时执行一次
- cron 模式
- 连续执行模式

### 第五步：设置发送消息方式

根据你的模式不同，你可以选择：

- 每次都发送同一个预设消息
- 第一次手动输入，后续自动沿用上次消息
- 连续执行模式下固定使用同一条预设消息

### 第六步：设置停止关键词

你可以按需配置一组“停止关键词”。

当后续 Codex 返回内容里包含任意一个关键词时，`codex_timer` 会自动结束当前循环任务。

例如：

```text
done
finished
任务完成
```

多个关键词可以用英文逗号分隔。

如果留空不填，则表示关闭这个功能。

## 实际会生成什么文件

在任意项目里运行 `codex_timer` 之后，会在当前项目生成：

```text
your-project/
  .codex_timer/
    config.json
    logs/
      timer.log
    response_YYYYMMDD_HHMMSS.txt
```

这意味着每个项目都拥有自己的：

- 配置
- 日志
- 响应保存文件

不同项目之间不会互相污染。

而用户全局偏好（比如界面语言）会单独保存在：

```text
~/.codex_timer/preferences.json
```

## 运行模式说明

### 固定间隔模式

按秒、分钟或小时重复执行。

适合：

- 定时检查状态
- 周期性跟进任务
- 轻量级自动化轮询

### Cron 模式

支持简化版五段 cron 表达式，例如：

```text
*/30 * * * *
0 */2 * * *
0 9 * * *
```

适合：

- 每天固定时间执行
- 每小时或每几小时执行
- 有明确计划时间点的自动任务

### 连续执行模式

这个模式会一直使用同一条预设消息。

每次任务执行结束后，等待 3 秒，然后自动再次发送这条消息开始下一轮。

适合：

- 持续推进型任务
- 类似流水线的自动循环
- “一直跑，直到我手动停止”为止的场景

### 停止关键词

这不是一种单独的调度模式。

它是一个可选的“自动结束条件”，可以和任何调度模式一起使用。

只要 Codex 输出命中了你配置的任意关键词，当前自动循环就会立即结束。

## 会话筛选逻辑

`codex_timer` 会读取本机的：

```text
~/.codex/sessions/
```

但不会把所有项目的会话都列出来。

它只保留满足下面条件的会话：

```text
session.cwd === 当前工作目录
```

也就是说，你在 A 项目运行时，不会看到 B 项目的会话。

## 常见问题

### 1. 执行 `codex_timer` 提示找不到命令

通常是因为：

- 没有全局安装
- 或者 npm 的全局 bin 目录没有加入 `PATH`

你可以先重新安装：

```bash
npm i -g codex-timer
```

### 2. 界面语言不对

你可以随时执行：

```bash
codex_timer -lang
```

或者直接指定：

```bash
codex_timer -lang zh
codex_timer -lang en
```

### 3. 执行时提示找不到 `codex`

`codex_timer` 依赖本机已经安装的 `codex` CLI。

如果 `codex` 命令本身不能运行，`codex_timer` 也无法工作。

### 4. 当前目录没有列出任何会话

一般有两种原因：

- 这个目录以前没有对应的 Codex 会话
- 之前的会话是在另一个路径下产生的

这时直接创建一个新会话即可。

### 5. 最近消息预览看起来不对

消息预览来自本地 `~/.codex/sessions` 里的 JSONL 文件。

如果未来 Codex 的 session 文件结构发生变化，预览提取逻辑可能需要同步调整。

### 6. 为什么任务没有自动停止

你可以检查：

- 是否配置了停止关键词
- Codex 的返回内容里是否真的包含这些关键词
- 关键词是否按子串匹配成功

如果没有命中任何关键词，循环会正常继续执行。
